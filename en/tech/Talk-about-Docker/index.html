What is Docker

This paragraph is copied from the Docker official website:

Docker uses the Go language launched by Google for development and implementation, based on the Linux kernel cgroup, namespace , and [Union FS](https://en.wikipedia .org/wiki/Union_mount) and other technologies, which encapsulate and isolate processes, belong to operating system level virtualization technology. Since the isolated process is independent of the host and other isolated processes, it is also called a container. The initial implementation is based on LXC. Since version 0.7, LXC has been removed and the self-developed [libcontainer](https://github.com/docker/ libcontainer), starting from 1.11, it further evolved to use runC and containerd.

Docker has carried out further encapsulation on the basis of containers, from file system, network interconnection to process isolation, etc., which greatly simplifies the creation and maintenance of containers. This makes Docker technology more portable and faster than virtual machine technology.

It’s too complicated to understand, isn’t it? A simple example to illustrate, in the past, what was the process for software companies to release products to customers for deployment and launch?

  1. Developers complete the development, package and submit for testing
  2. Testers set up a test environment for testing, and return to development if there are bugs
  3. The developer will verify the bug, if it can be reproduced, modify the bug and resubmit the test
  4. If developers cannot reproduce, they need to communicate with testers to confirm and investigate environmental factors (generally this cycle will be very scary)
  5. After the test is completed, the developer will package the version and submit it to the on-site implementation staff
  6. The implementer gets the package** and deploys the UAT environment**, submits it for user testing after deployment, and returns to the implementer if there is a bug
  7. The implementer needs to check: user operation**, deployment process, UAT environment and other factors**, if it cannot be resolved, then return to the developer for verification
  8. The developer verifies, if the bug can be reproduced, modify the bug, and repeat steps 4, 5, 6
  9. The implementer deploys to the production environment and conducts the final test. If there is a bug, repeat steps 7 and 8

It can be seen that in the entire software life cycle, environment deployment and environment inconsistency will cause big problems. Docker solves this problem perfectly. If Docker is used in the entire process, the process will be greatly simplified, such as:

  1. The developer completes the development and marks as a Docker image to submit for testing
  2. The tester starts the mirror for testing, and returns to development if there is a bug
  3. The developer verifies the bug, if it can be reproduced, modify the bug and resubmit the test (after having Docker, if the developer follows the test procedure, 99% of the problem can be reproduced)
  4. After the test is completed, the developer marks it as a Docker image and submits it to the on-site implementation staff
  5. The implementer starts the mirror, the user conducts a UAT test, and returns to the implementer if there is a bug
  6. The implementer needs to check: user operation (99% of the problems are improper user operation or insufficient test case coverage by the tester after Docker is available). If it cannot be solved, return it to the developer for verification
  7. The developer verifies, if the bug can be reproduced, modify the bug, and repeat steps 4, 5, 6
  8. The implementer starts the mirror and performs the final test. If there is a bug, repeat steps 7 and 8

**Docker’s external slogan is: once package, run everywhere, and the operating environment is exactly the same. **When you finish the development, you package Tomcat, Java environment and web application into an image for packaging, then this image can be copied everywhere, as long as the machine equipped with Docker can extract the image and ** run directly and packaged with you For an application with the same environment, databases, middleware, and Web applications can all be mirrored. **

More importantly, compared with traditional virtual machines:

  1. Docker starts in seconds
  2. Docker’s disk usage is MB
  3. Docker’s performance is close to native
  4. Docker’s single machine supports thousands of containers

Install Docker under Mac OS

Installing Docker under Mac OS is very simple. After registering on the official website directly click on Get Docker to download the installation file. However, for some well-known reasons, if you want to extract mirrors elegantly, you also need to configure Registry mirrors.

https://registry.docker-cn.com

Install Docker under CentOS

Installing Docker under CentOS is equally simple, but you need to determine the kernel version first. Docker requires the kernel version of the CentOS system to be higher than 3.10

  1. First check the kernel version

​ uname -r

  1. Download and install, here you need to pay attention, after March 2017, docker is divided into two branch versions on the original basis: Docker CE and Docker EE. Docker CE is the community free version, and Docker EE is the enterprise version. It emphasizes security, but it needs to be used for a fee.

    Install some necessary system tools, yum-util provides the yum-config-manager function, and the other two are dependent on the devicemapper driver:

    sudo yum install -y yum-utils device-mapper-persistent-data lvm2

    Because of some well-known reasons, you need to set up a domestic yum source:

    sudo yum-config-manager --add-repo http://mirrors.aliyun.com/docker-ce/linux/centos/docker-ce.repo

    Update yum cache:

    sudo yum makecache fast

    Install Docker:

    sudo yum -y install docker-ce

    Start Docker service:

    sudo systemctl start docker

Use of Docker

If it is only used normally, the commands that need to be understood are very limited, the most commonly used are the following five

List the mirror list

Use docker images to list the images on the local host

Find mirror

Use the docker search [mirror name] command to search for mirrors, such as searching for the mirrors I packaged

When searching, you can either directly search for the mirror name, or search the author name/mirror name, it will be sorted by the number of STARS, and OFFICIAL indicates that it is an official mirror.

Get a new mirror

Use the docker pull [image name] command to get the image

If there is no version number, the latest version of the mirror will be downloaded automatically, or if you know the name of the mirror specifically, you can directly use docker run [mirror name] This time the warehouse will automatically pull if there is no such mirror.

Delete mirror

Use the docker image rm [IMAGE ID] command to delete the image, such as deleting the image obtained above

Run the mirror

Use the docker run [mirror name] command to start the mirror. It is not recommended to use it directly here, because in general, you need to configure the port mapping and disk mapping between the host and the mirror when starting up. It is recommended to use **Dockerfile ** to run, put it in the Docker build and talk about it together

View the running status of the mirror

Use the docker stats command to view the running status of the image

Dockerfile

In another sentence, Dockerfile can be understood as a BAT file on Windows (but there are still differences in fact), which can be regarded as a set of dockerfile configuration statements executed sequentially, but since it is said that Docker builds, it needs to be said briefly. The composition of Docker, theoretical knowledge is not mentioned, to understand it this way, the composition of Docker is like an onion, layer by layer, the core is the original container, it can be empty, or it can be a basic Java environment , Just like the picture below

This leads to the first command of the Dockerfile

FROM, specify the base image, for example, it can be written as FROM java:8, this is the version that specifies the base image as java and tag is 8, corresponding to jdk is 1.8

EXPOSE, specify the mirror port number, for example, it can be written as EXPOSE 8080, indicating that the external port of this mirror is 8080

VOLUME, specify the folder for data persistence. **It should be highlighted here that docker mirroring runs like redis. Once the mirroring stops, any data changes that occur inside will not be saved! ! ! Therefore, you need to specify a folder, this folder is mapped to the host, and all the data that needs to be saved should be placed in this folder! ! ! **This can be written as VOLUME /tmp, because the default working directory of the embedded Tomcat container used by Spring Boot is /tmp

ADD, put the content of the specified path into the specified path of the mirror, this is mainly used to throw the packaged web file into the mirrored web container, such as this sentence ADD jmall-admin.jar /app.jar It will throw my packaged Spring Boot file into the root directory of the container and rename it to app.jar

RUN, this is easy to understand, is to execute the docker command, corresponding to the above docker run

ENTRYPOINT, specify the default entry point of the mirror, such as ENTRYPOINT ["java","-jar","/app.jar"], this is particularly familiar java -jar /app.jar

Use Dockerfile

Using Dockerfile is very simple. The easiest way is to go directly to the directory at the same level of Dockerfile and run the command docker build directly. The created image will be directly placed in your local warehouse. You can use the docker images command to view

Submit your mirror

After the image is done, you can directly submit the image file to the private server or docker hub to share with others. Submitting the image is also very simple. I submitted it directly with IDEA. If it is on the server, you can also use the command

First log in to docker hub

docker login

After pressing enter, you will be asked to enter your username and password

You can submit it when you see Login Succeeded, docker push wellmoonloft/jmall here needs attention, your name in the front must be the same as your login user name, and due to well-known reasons… The submission process is very long, so it is strongly recommended to submit it with a tool, at least there is a progress bar to see.

Save the image as a file and copy it to others

Sometimes the image that we packaged belongs to the internal system, which is not convenient to push to the public network, or if our company does not have a private server, we need to save the image to others as a tar file and copy it to others. The image storage is also very simple. Command docker save -o [file name to be saved] [image file to be saved or IMAGE ID], after completion, there will be a tar package in the path where you execute the command, this package can be directly copied to others. It should be noted that if it is Cent OS or Mac OS, pay attention to the authorization of the path

Import the image into the local warehouse

You can’t directly use the mirrored tar file you received from others. You need to import it into your local warehouse. It’s also very simple. Execute the command docker load <[file name], corresponding to the above file, you should write docker load < jmall.tar

Docker Compose

If you just run a simple single application, this is enough, but in production, multiple applications often work together to complete the construction of a system. If you add load balancing, you may have to start or stop multiple mirrors at the same time. And unified management, there is a very useful mechanism in Docker called Docker Compose to solve this problem, here we need to introduce two familiar terms.

-Service (service): a single container image, which can be a database or a web application
-Project (project): A complete business unit composed of a set of associated application containers (service)

Don’t talk about concepts, look at examples:

version: '2' This is not just scribbled, this version is the version of Docker Compose installed on the host

services is a fixed wording, indicating that the following are all mirror images

The name of the jmall-adminservice, which corresponds to the image name in the following image

ports, the mirror port is specified as 8080 when building the mirror, but in fact it cannot be accessed when the mirror is run directly, because the port thrown by the mirror is not mapped with the host port , Here is to map the 8080 of the host machine to the 8080 in the image, of course, if you don’t use Docker Compose, you can also directly map it with docker run -p 8080:8080

environment is to force the content of the Spring Boot configuration file to be changed, here I am writing to specify the development environment

Use Docker Compose

It is also very simple to use Dokcer Compose. Enter the same level path of Dokcer Compose and use the command docker-compose up
